## Задача 1. Инициализация репозитория и первый коммит

### Контекст

Тебе дали задачу: написать утилиту `devtool`. Первый шаг — создать проект и сразу поставить его под контроль версий. Хорошая привычка: репозиторий инициализируют *до* того, как написана первая строка кода, а не после.

### Задание

1. Создай папку `devtool/` и перейди в неё.
2. Инициализируй Git-репозиторий.
3. Создай файл `main.py` со следующим содержимым:

```python
def count_lines(filepath):
    """Возвращает количество строк в файле."""
    with open(filepath, 'r', encoding='utf-8') as f:
        return sum(1 for _ in f)

if __name__ == '__main__':
    import sys
    print(count_lines(sys.argv[1]))
```

4. Добавь файл в индекс и сделай первый коммит с осмысленным сообщением.

### Подсказка

```
mkdir, cd, git init
git add <файл>
git commit -m "..."
```

Сообщение коммита — в повелительном наклонении, на английском или русском: `"Add count_lines function"` или `"Добавить функцию подсчёта строк"`.

### Проверка

После коммита `git log --oneline` должен показать одну строку:

```
a1b2c3d Add count_lines function
```

А `git status` — чистое состояние:

```
On branch main
nothing to commit, working tree clean
```

### Разбор

```bash
mkdir devtool && cd devtool

# Инициализируем репозиторий — создаётся скрытая папка .git
git init

# Создаём main.py (содержимое выше)

# git add сообщает Git: "включи этот файл в следующий коммит"
git add main.py

# git commit фиксирует снимок — теперь это навсегда в истории
git commit -m "Add count_lines function"
```

**Что происходит внутри:** `git init` создаёт `.git/` — там хранится вся история, индекс и настройки. Без этой папки репозитория не существует.

> 💡 **Windows:** Если `git` не распознаётся — убедись, что Git установлен и добавлен в PATH. Скачать: https://git-scm.com/download/win

---

### ⚠️ Частые ошибки

**Ошибка 1: Коммит пустой — `nothing to commit`**

Забыли сделать `git add` перед `git commit`. Git не коммитит файлы автоматически — сначала нужно добавить в индекс.

```bash
# Неверно:
git commit -m "Add main.py"  # → error: nothing to commit

# Верно:
git add main.py
git commit -m "Add main.py"
```

**Ошибка 2: `git init` выполнен не в той папке**

Проверь где ты находишься:

```bash
pwd        # Linux/macOS
cd         # Windows — покажет текущую папку
ls -la     # должна быть папка .git
```

## Задача 2. Второй коммит: добавляем README

### Контекст

Проект без документации — не проект. `README.md` — первое, что видит любой разработчик (и ты сам через полгода). Добавляем его отдельным коммитом, потому что это логически самостоятельное изменение.

### Задание

1. Создай файл `README.md` в папке `devtool/` со следующим содержимым (можно адаптировать):

```markdown
# devtool

CLI-утилита для анализа текстовых файлов.

## Возможности
- Подсчёт строк в файле

## Использование
    python main.py path/to/file.txt
```

2. Добавь файл в индекс и закоммить *отдельным* коммитом.
3. Посмотри историю через `git log --oneline`.

### Подсказка

```
git add README.md
git diff --staged
git commit -m "..."
git log --oneline
```

### Проверка

После `git add README.md`, но до коммита — выполни `git diff --staged`. Увидишь:

```
diff --git a/README.md b/README.md
new file mode 100644
index 0000000..abc1234
--- /dev/null
+++ b/README.md
@@ -0,0 +1,8 @@
+# devtool
+...
```

`git log --oneline` после коммита покажет **два** коммита (новые сверху):

```
f4e5d6c Add README with project description
a1b2c3d Add count_lines function
```

### Разбор

```bash
# Создаём README.md с содержимым выше

# Добавляем только README — main.py уже в истории, трогать не надо
git add README.md

# Смотрим что войдёт в коммит — то что уже в индексе
git diff --staged   # или синоним: git diff --cached

# Обычный git diff сейчас ничего не покажет — все изменения уже в индексе
git diff            # пусто

git commit -m "Add README with project description"

# Смотрим историю в компактном формате
git log --oneline
```

**Три уровня Git и какой `diff` что показывает:**

```
Рабочая директория  →  git add  →  Индекс (staging)  →  git commit  →  История
        ↑                               ↑
   git diff                      git diff --staged
(ещё не в индексе)             (уже в индексе, войдёт в коммит)
```

Это различие критично когда ты, например, сделал `git add`, потом ещё что-то поправил в файле — `git diff` покажет правку после `add`, а `git diff --staged` покажет что реально попадёт в коммит.

**Почему отдельный коммит?** Каждый коммит — один логический шаг. Если завтра окажется, что README написан неверно — его можно откатить, не трогая код. Смешивать несвязанные изменения в один коммит — плохая практика.

---

### ⚠️ Частые ошибки

**Ошибка 1: Случайно добавили оба файла через `git add .`**

`git add .` добавляет *всё* в текущей папке. Для обучения лучше добавлять явно. Если уже добавил лишнее — убери из индекса:

```bash
git restore --staged main.py   # убрать файл из индекса (не удаляет сам файл)
```

**Ошибка 2: Непонятное сообщение коммита**

`"fix"`, `"update"`, `"asdfgh"` — плохие сообщения. Хорошее сообщение отвечает на вопрос: *что сделано и зачем?* Например: `"Add README with usage instructions"`.

## Задача 3. Изучение истории: `git log` и `git show`

### Контекст

История коммитов — это журнал проекта. Умение читать её экономит часы отладки. Сейчас у тебя два коммита — самое время разобраться как смотреть, что именно изменилось в каждом.

### Задание

1. Посмотри полную историю командой `git log` (выйти — клавиша `q`).
2. Посмотри компактную историю: `git log --oneline`.
3. Скопируй хэш первого коммита (7 символов достаточно) и выполни `git show <hash>`.
4. Изучи вывод: найди автора, дату, сообщение и diff изменений.

### Подсказка

```
git log
git log --oneline
git show <первые 7 символов хэша>
```

### Проверка

`git show <hash>` выведет что-то вроде:

```
commit a1b2c3d...
Author: Your Name <you@example.com>
Date:   Mon Jan 1 12:00:00 2024 +0300

    Add count_lines function

diff --git a/main.py b/main.py
new file mode 100644
index 0000000..abc1234
--- /dev/null
+++ b/main.py
@@ -0,0 +1,8 @@
+def count_lines(filepath):
...
```

Зелёные строки с `+` — что было добавлено.

### Разбор

```bash
# Полный лог: хэш, автор, дата, сообщение
git log

# Компактный: хэш (7 символов) + сообщение — удобно для обзора
git log --oneline

# Показать конкретный коммит целиком
git show a1b2c3d

# Показать только изменения без метаданных
git show a1b2c3d --stat   # только список файлов
```

**Как читать diff:** строки с `+` — добавленные, с `-` — удалённые. `@@` показывает номера строк в файле.

---

### ⚠️ Частые ошибки

**Ошибка 1: Застрял в `git log` — не могу выйти**

`git log` открывает pager (обычно `less`). Выход — клавиша `q`. Прокрутка — стрелки или `j`/`k`.

Если хочешь отключить pager навсегда:
```bash
git config --global core.pager cat
```

**Ошибка 2: Хэш не распознаётся**

Скопировал лишний пробел или символ. Используй минимум 4 символа хэша — Git найдёт сам, если он уникален. Или скопируй прямо из вывода `git log --oneline`.

## Задача 4. Откат изменений: `git restore`

### Контекст

Классическая ситуация: пошёл "быстро поправить" функцию, что-то сломал, а сохранил уже несколько раз. Именно для этого существует `git restore` — он возвращает файл к состоянию последнего коммита.

### Задание

1. Открой `main.py` и "случайно" сломай его — например, удали две строки из функции или добавь синтаксическую ошибку.
2. Сохрани файл.
3. Убедись что `git status` показывает изменения.
4. Откати `main.py` к состоянию последнего коммита через `git restore`.
5. Проверь что файл вернулся в исходное состояние.

### Подсказка

```
git status
git diff main.py
git restore main.py
```

### Проверка

После `git restore` команда `git status` снова покажет:

```
nothing to commit, working tree clean
```

А `cat main.py` выведет оригинальный код без изменений.

### Разбор

```bash
# Ломаем файл (пример — удаляем строку с return)

# Смотрим что изменилось
git status          # покажет: modified: main.py
git diff main.py    # покажет конкретные строки красным/зелёным

# Откатываем к последнему коммиту — изменения ПОТЕРЯЮТСЯ безвозвратно
git restore main.py

# Проверяем
git status          # nothing to commit
```

**Важно:** `git restore` уничтожает несохранённые изменения. Нет способа их вернуть после этой команды. Если изменения нужно сохранить на время — используй `git stash` (это тема следующих модулей).

> 💡 **Старый синтаксис:** В Git до 2.23 вместо `git restore` использовали `git checkout -- main.py`. Обе команды делают одно и то же. `git restore` — современный способ.

---

### ⚠️ Частые ошибки

**Ошибка 1: Восстановил файл, которого нет в индексе**

Если файл новый (ни разу не коммитился) — `git restore` его не тронет, потому что Git не знает к какому состоянию возвращать. Для удаления нового файла используй обычный `rm`.

**Ошибка 2: Хотел откатить staging, а не рабочую директорию**

Если уже сделал `git add` и хочешь убрать файл из индекса (но не удалять изменения):

```bash
git restore --staged main.py   # убирает из индекса, файл остаётся изменённым
git restore main.py            # откатывает изменения в файле
```

## Итоги модуля

Ты освоил:

| Команда | Что делает |
|---|---|
| `git init` | Инициализирует репозиторий |
| `git add <file>` | Добавляет файл в индекс (staging area) |
| `git commit -m "..."` | Фиксирует снимок состояния |
| `git status` | Показывает состояние рабочей директории и индекса |
| `git log` / `git log --oneline` | История коммитов |
| `git show <hash>` | Показывает конкретный коммит |
| `git diff` | Изменения в рабочей директории, ещё не добавленные в индекс |
| `git diff --staged` | Изменения в индексе, которые войдут в следующий коммит |
| `git restore <file>` | Откатывает файл к последнему коммиту |

**Главная идея модуля:** цикл `edit → stage → commit` — основа работы с Git. Каждый коммит — это точка, к которой можно вернуться. Чем осмысленнее коммиты, тем легче работать с историей.
