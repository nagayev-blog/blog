# Модуль 1.2 — `.gitignore` и структура проекта

> **Контекст модуля:** Проект `devtool` растёт — появляются зависимости, тесты, виртуальные окружения. Git видит всё это и предлагает закоммитить. Но не всё должно попадать в репозиторий: скомпилированные файлы, секреты, мусор от IDE. В этом модуле ты научишься контролировать что именно отслеживает Git — и коммитить избирательно.

**Что ты сделаешь в этом модуле:**
- Увидишь проблему: Git хочет закоммитить мусор
- Создашь `.gitignore` и разберёшься с его синтаксисом
- Освоишь `git add -p` — частичное добавление изменений
- Поймёшь разницу между рабочей директорией и индексом через `git status` и `git diff`

---

## Задача 1. Воспроизведи проблему: мусор в репозитории

### Контекст

Когда разработчик начинает работать над проектом, первым делом он настраивает **окружение**: устанавливает зависимости, создаёт виртуальное окружение, запускает тесты. Всё это порождает файлы и папки, которые Git видит и предлагает закоммитить — но которые в репозитории делать нечего:

- `requirements.txt` — список зависимостей, нужен в репозитории ✅
- `__pycache__/` — байткод Python, генерируется автоматически при импорте модулей ❌
- `venv/` — виртуальное окружение, весит сотни мегабайт и у каждого своё ❌
- `.env` — секреты и токены, вообще нельзя коммитить ❌

Сейчас воспроизведём эту ситуацию и посмотрим на неё глазами Git.

### Задание

1. Перейди в папку `devtool/`.
2. Создай папку `tests/` и файл `tests/test_main.py` с содержимым:

```python
from main import count_lines, count_words

def test_count_lines():
    assert count_lines('tests/sample.txt') == 3

def test_count_words():
    assert count_words('tests/sample.txt') == 6
```

3. Создай тестовый файл `tests/sample.txt`:

```
hello world
foo bar
baz qux
```

4. Добавь `requirements.txt`:

```
pytest==8.1.0
```

5. Спровоцируй появление `__pycache__` — он создаётся при **импорте** модуля (не при запуске скрипта напрямую):

```bash
python -c "import main"
```

> 💡 **macOS/Linux:** если команда `python` не найдена — попробуй `python3 -c "import main"`

6. Выполни `git status` и изучи вывод — что Git предлагает добавить?

### Подсказка

```
git status
```

### Проверка

`git status` покажет что-то вроде:

```
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        __pycache__/
        requirements.txt
        tests/
```

Git видит папку `__pycache__/` и хочет её отслеживать. Это проблема.

### Разбор

```bash
# 1. Создаём папку tests/ и файлы внутри
mkdir tests
# tests/test_main.py и tests/sample.txt — создай вручную с содержимым из задания

# 2. Создаём requirements.txt
echo "pytest==8.1.0" > requirements.txt

# 3. Импортируем модуль — Python создаёт __pycache__/
python -c "import main"
# или на macOS/Linux если python не найден:
python3 -c "import main"

# 4. Смотрим что Git видит
git status
```

Вывод `git status` будет примерно таким:

```
Untracked files:
        __pycache__/      ← сгенерирован Python автоматически
        requirements.txt  ← нужен в репозитории
        tests/            ← нужен в репозитории
```

Обрати внимание: Git не различает "нужные" и "ненужные" файлы — он просто видит всё новое. Если сейчас выполнить `git add .` — `__pycache__/` попадёт в историю вместе с остальным.

**Почему `__pycache__` — проблема?** Это скомпилированные байткод-файлы Python. Они генерируются автоматически при импорте, зависят от версии Python и платформы. У тебя они одни, у коллеги — другие. Коммитить их бессмысленно: засоряют историю и создают ложные конфликты при слиянии.

---

### ⚠️ Частые ошибки

**Ошибка 1: `__pycache__` не появился**

`__pycache__` создаётся при **импорте** модуля, а не при запуске скрипта через `python main.py`. Убедись что выполнил именно:

```bash
python -c "import main"
# или
python3 -c "import main"
```

На macOS команда `python` может указывать на Python 2 или отсутствовать — используй `python3`.

**Ошибка 2: `git status` ничего не показывает**

Возможно, находишься не в папке репозитория. Проверь:

```bash
pwd          # Linux/macOS
cd           # Windows
git status   # должен показывать ветку и файлы
```

## Задача 2. Создаём `.gitignore`

### Контекст

`.gitignore` — файл с правилами: какие файлы и папки Git должен игнорировать. Один раз настроил — и больше не думаешь об этом.

### Задание

1. Создай файл `.gitignore` в корне проекта (`devtool/.gitignore`) со следующим содержимым:

```
# Python
__pycache__/
*.pyc
*.pyo
*.pyd

# Виртуальные окружения
venv/
.venv/
env/

# Секреты и конфигурация
.env
.env.*

# IDE
.idea/
.vscode/
*.swp
```

2. Выполни `git status` снова — убедись что `__pycache__` пропал из списка.
3. Добавь `.gitignore` и `requirements.txt` в индекс и закоммить.

### Подсказка

```
git status
git add .gitignore requirements.txt
git diff --staged
git commit -m "..."
```

### Проверка

После создания `.gitignore` команда `git status` не должна показывать `__pycache__/`:

```
Untracked files:
        .gitignore
        requirements.txt
        tests/
```

После коммита — `git log --oneline` покажет новый коммит.

### Разбор

```bash
# Создаём .gitignore (содержимое выше)

git status
# __pycache__/ исчез — Git теперь его игнорирует

git add .gitignore requirements.txt
git diff --staged   # проверяем что войдёт в коммит

git commit -m "Add .gitignore and requirements.txt"
```

**Синтаксис `.gitignore`:**

| Паттерн | Что игнорирует |
|---|---|
| `__pycache__/` | Папку с таким именем (где угодно в проекте) |
| `*.pyc` | Все файлы с расширением `.pyc` |
| `.env` | Файл `.env` в корне |
| `# комментарий` | Ничего — это комментарий |
| `!important.pyc` | Исключение: НЕ игнорировать этот файл |

> 💡 **Готовые шаблоны:** GitHub поддерживает коллекцию `.gitignore` для разных языков и фреймворков: https://github.com/github/gitignore. Для Python-проектов можно взять `Python.gitignore` как базу.

---

### ⚠️ Частые ошибки

**Ошибка 1: `.gitignore` не работает — файл всё равно отслеживается**

Скорее всего, файл уже был добавлен в Git до создания `.gitignore`. Правила игнорирования не применяются к уже отслеживаемым файлам. Решение:

```bash
git rm --cached __pycache__/   # убрать из отслеживания, не удаляя с диска
# или для папки:
git rm -r --cached __pycache__/
```

**Ошибка 2: Случайно закоммитили `.env` с секретами**

Срочно: удали файл из истории (это нетривиально — `git filter-branch` или `git-filter-repo`), смени все скомпрометированные ключи. Проще — не допустить: добавляй `.env` в `.gitignore` до первого коммита.

## Задача 3. Частичное добавление: `git add -p`

### Контекст

В реальной работе часто бывает так: сидишь над одной задачей и попутно замечаешь что-то ещё — поправляешь опечатку в комментарии, улучшаешь docstring соседней функции. В итоге один файл содержит правки **по двум разным причинам**. Коммитить их одним куском — плохо: история становится грязной, сложнее понять что и зачем менялось. `git add -p` позволяет выбрать конкретные куски изменений внутри файла и разложить их по разным коммитам.

### Задание

Сымитируем реальную ситуацию: ты работаешь над добавлением новой функции `count_chars`, но по ходу замечаешь что у `count_lines` нет docstring — и добавляешь его заодно. Два несвязанных изменения в одном файле, две разные причины.

1. Открой `main.py` и внеси **оба** изменения сразу:

   **В начале файла** — добавь docstring к существующей функции `count_lines`:

```python
def count_lines(filepath):
    """Возвращает количество строк в файле."""  # <- добавить эту строку
    with open(filepath, 'r', encoding='utf-8') as f:
        return sum(1 for _ in f)
```

   **В конце файла** — добавь новую функцию `count_chars`:

```python
def count_chars(filepath):
    """Возвращает количество символов в файле."""
    with open(filepath, 'r', encoding='utf-8') as f:
        return sum(len(line) for line in f)
```

2. Запусти `git add -p main.py` — Git покажет два отдельных куска (hunk), по одному на каждое изменение.
3. Добавь в индекс **только** docstring к `count_lines`, пропусти `count_chars`.
4. Проверь через `git diff --staged` и `git diff` что в индексе ровно то что нужно.
5. Закоммить первый кусок.
6. Добавь `count_chars` и закоммить отдельным коммитом.

### Подсказка

```
git add -p main.py
# в интерактивном режиме:
# y — добавить этот кусок в индекс
# n — пропустить этот кусок
# s — разбить на меньшие куски
# q — выйти
git diff --staged   # что войдёт в коммит
git diff            # что осталось вне индекса
git commit -m "..."
```

### Проверка

После `git add -p` но до первого коммита:

```bash
git diff --staged
# показывает только docstring к count_lines

git diff
# показывает только функцию count_chars
```

После двух коммитов `git log --oneline`:

```
a1b2c3d Add count_chars function
b2c3d4e Add docstring to count_lines
...
```

Два логически разных изменения — два отдельных коммита. Именно так выглядит чистая история.

### Разбор

```bash
# Вносим оба изменения в main.py:
# - docstring к count_lines в начале файла
# - новую функцию count_chars в конце файла

git add -p main.py
# Git находит два hunk — они далеко друг от друга в файле,
# поэтому Git разбивает их автоматически, без команды s

# Первый hunk — docstring к count_lines (начало файла):
# Stage this hunk [y,n,q,a,d,s,?]? y   <- добавляем

# Второй hunk — функция count_chars (конец файла):
# Stage this hunk [y,n,q,a,d,s,?]? n   <- пропускаем

git diff --staged   # только docstring — count_chars здесь нет
git diff            # только count_chars — docstring уже в индексе

git commit -m "Add docstring to count_lines"

# Теперь добавляем второй кусок
git add main.py
git commit -m "Add count_chars function"
```

**Почему Git сам разбил на два hunk?** Изменения находятся далеко друг от друга в файле — Git видит их как независимые куски. Если бы правки стояли рядом, попали бы в один hunk и пришлось бы вводить `s` для разбивки.

**Это не академический пример** — именно так выглядит повседневная работа. Поправил опечатку попутно с основной задачей? Два коммита. Отрефакторил функцию и заодно добавил новую? Два коммита. `git add -p` — инструмент дисциплины, который делает историю читаемой.

---

### ⚠️ Частые ошибки

**Ошибка 1: Git показывает один hunk вместо двух**

Изменения оказались слишком близко в файле — Git объединил их. Введи `s` (split) — Git попробует разбить. Если не получается (правки вплотную без пустых строк) — используй `e` (edit) для ручного редактирования hunk.

**Ошибка 2: Запутался в интерактивном режиме, хочу начать заново**

Введи `q` (quit) — выйдет без добавления текущего куска. Всё что уже ответил `y` — останется в индексе. Сбросить индекс полностью и начать заново:

```bash
git restore --staged main.py   # убрать всё из индекса
git add -p main.py             # начать сначала
```


## Задача 4. `git status` и `git diff` — понять разницу

### Контекст

К этому моменту ты уже несколько раз использовал `git status` и `git diff`. Пора разобраться чётко: что именно каждая из них показывает и когда какую применять.

### Задание

1. Внеси любое изменение в `main.py` (например, добавь комментарий в начало файла: `# devtool - text analysis utility`).
2. Выполни `git status` — запомни вывод.
3. Выполни `git diff` — изучи вывод.
4. Выполни `git add main.py`.
5. Выполни `git status` снова — что изменилось?
6. Выполни `git diff` — что изменилось?
7. Выполни `git diff --staged` — что видишь?
8. Закоммить изменение.

### Подсказка

```
git status
git diff
git add main.py
git status
git diff
git diff --staged
git commit -m "..."
```

### Проверка

Заполни мысленно таблицу — после каждого шага ты должен понимать почему вывод именно такой:

| Момент | `git status` | `git diff` | `git diff --staged` |
|---|---|---|---|
| После правки, до `add` | modified (red) | показывает изменение | пусто |
| После `git add` | modified (green) | пусто | показывает изменение |
| После `git commit` | clean | пусто | пусто |

### Разбор

```bash
# Правим main.py

git status
# → Changes not staged for commit:
# →   modified: main.py   (красный — в рабочей директории, не в индексе)

git diff
# → показывает что изменилось в файле (рабочая директория vs индекс)

git add main.py

git status
# → Changes to be committed:
# →   modified: main.py   (зелёный — в индексе, войдёт в коммит)

git diff
# → ничего! изменения теперь в индексе, рабочая директория == индекс

git diff --staged
# → показывает что в индексе (индекс vs последний коммит)

git commit -m "Add file description comment"

git status
# → nothing to commit, working tree clean
```

**Схема трёх состояний:**

```
[Рабочая директория] → git add → [Индекс] → git commit → [История]
         ↑                           ↑
    git diff                  git diff --staged
  (что изменилось,           (что войдёт
   но не добавлено)           в коммит)
```

---

### ⚠️ Частые ошибки

**Ошибка 1: `git diff` пустой, хотя файл изменён**

Значит, изменения уже в индексе (после `git add`). Используй `git diff --staged` чтобы их увидеть.

**Ошибка 2: `git status` показывает файл дважды**

```
Changes to be committed:
    modified: main.py   ← эта версия в индексе

Changes not staged for commit:
    modified: main.py   ← новые правки после git add
```

Это нормально — ты добавил файл в индекс, потом ещё что-то поправил. В коммит попадёт версия из индекса, а не последняя. Чтобы обновить индекс — снова `git add main.py`.

## Итоги модуля

Ты освоил:

| Команда | Что делает |
|---|---|
| `.gitignore` | Файл с правилами игнорирования — Git не видит указанные файлы |
| `git rm --cached <file>` | Убирает файл из отслеживания (не удаляет с диска) |
| `git add -p <file>` | Интерактивное частичное добавление — выбираешь конкретные куски |
| `git status` | Показывает состояние рабочей директории и индекса |
| `git diff` | Изменения в рабочей директории, ещё не добавленные в индекс |
| `git diff --staged` | Изменения в индексе, которые войдут в следующий коммит |

**Главная идея модуля:** репозиторий должен содержать только то, что нужно для работы проекта. Всё остальное — в `.gitignore`. А `git add -p` даёт контроль над тем, что именно попадает в каждый коммит — даже если изменений много и все в одном файле.
